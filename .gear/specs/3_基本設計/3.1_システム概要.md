# システム概要

> バージョン: 2 | 更新日時: 2026/3/21 12:21:21

**sections:**

### システム構成概要

- **content:** mDNSプロキシシステムは、ネットワークセグメント跨ぎのmDNS名前解決を実現する分散型プロキシシステムです。各セグメントに配置されたプロキシが相互にmDNSレコードを交換し、統合されたレコード情報でmDNS応答を行います。システムはRaspberry Pi Pico（MicroPython）とWindows EXE（Python）の2つのプラットフォームで動作し、SQLite3データベースによる7つのDB構成（static_hosts、self_records、other_records、other_proxies、merged_records、system_config、system_logs）で情報管理を行います。30秒間隔のメインループでプロキシ発見、データ同期、名前解決、mDNS応答の一連処理を実行します。

### 使用技術スタック

- **content:** 【プラットフォーム・言語】MicroPython（Raspberry Pi Pico版）、Python（Windows EXE版、PyInstallerでビルド）【データベース】SQLite3（軽量組み込みDB）【ネットワーク通信】UDP/5353（mDNS標準プロトコル）、HTTP/TCP80（プロキシ間データ交換）【名前解決】DNSプロトコル直接実装（dig、Resolve-DnsName、ping、arpingコマンド相当の自立型名前解決機能）【データ形式】JSON（プロキシ間データ交換）、INI（設定ファイル）【認証方式】トークン認証（接頭語+MACアドレス）【Wi-Fi接続】Raspberry Pi Pico標準Wi-Fi機能【管理インタフェース】CLI対話メニュー【ログ】ローカルファイル記録（INFO/ERRORレベル）【将来拡張】TLS暗号化、Web UI、REST API対応検討

### システム間連携

- **content:** 【プロキシ発見】固定IPリスト（システム.ini設定）による能動発見と、発信トークン受信による受動発見を併用し、other_proxiesテーブルで管理【データ同期】HTTP（TCP/80）でJSON形式のレコードデータを交換。static_hostsは全プロキシ間で最新タイムスタンプ優先で共有、各プロキシのself_recordsは他のother_recordsに複製【認証連携】HTTP通信時にヘッダでトークン（接頭語+MACアドレス）を送信し、受信側で検証【エラー処理】外部プロキシ接続失敗時はスキップして次サイクルで再試行、単一障害がシステム全体に影響しない設計【データ整合性】タイムスタンプ最新優先のマージルールで重複レコードを解決、merged_recordsテーブルで統合管理

### データフロー概要

- **content:** 【入力フロー】設定ファイル（検索ホスト名.ini、システム.ini）→static_hostsテーブル初期化→自立型名前解決実行（DNS依存なし）→self_recordsテーブル更新【同期フロー】プロキシ発見→static_hosts共有（最新タイムスタンプ優先）→外部プロキシからself_records相当データ取得→other_recordsテーブル更新【統合フロー】self_recordsとother_recordsをタイムスタンプ最新優先でマージ→merged_recordsテーブル作成【出力フロー】mDNS問い合わせ受信→merged_recordsテーブル検索→mDNS応答送信【管理フロー】CLI操作→設定変更→system_configテーブル・INIファイル更新→システム反映【クリーンアップフロー】TTL管理→期限切れレコード削除→データベース最適化【ログフロー】システム動作・エラー情報→system_logsテーブル記録

### セキュリティ概要

- **content:** 【認証方式】プロキシ間通信でトークン認証（接頭語+MACアドレス）を実装。HTTPヘッダでトークン送信し、受信側で検証【通信セキュリティ】現在はHTTP平文通信のため盗聴・改ざんリスクあり。運用ネットワークの信頼性確保が前提、将来的にTLS導入を推奨【アクセス制御】管理インタフェース（CLI）のアクセス制御は未定義（要確認事項）【データ保護】ローカルSQLite3データベースとINI設定ファイルの保護は運用環境に依存【名前解決セキュリティ】DNS依存を排除し自立型名前解決を採用することで、DNSサーバー障害や改ざんリスクを軽減【セキュリティ制約】HTTP平文通信による機密性の欠如、トークン固定化によるリプレイ攻撃リスク【推奨対策】信頼できるネットワーク環境での運用、将来的なTLS暗号化導入、定期的なトークン更新検討

### インフラ構成概要

- **content:** 【Raspberry Pi Pico環境】MicroPython実行環境、Wi-Fi接続によるネットワーク参加、ローカルストレージ（SQLite3、設定ファイル、ログファイル）、メモリ制約に注意した軽量実装、自立型名前解決機能によるDNSサーバー非依存【Windows環境】PyInstallerによるEXE配布、インストーラによるサービス化対応、有線・無線LAN接続対応、充分なストレージ・メモリリソース、dig/Resolve-DnsName/ping/arping相当の名前解決コマンド内蔵【共通要件】UDP/5353ポート（mDNS受信）、TCP/80ポート（HTTP通信）のファイアウォール開放、各セグメントに最低1台のプロキシ配置、DNSサーバー設定不要（自立型名前解決）【運用要件】24時間365日稼働想定、自動再接続機能による接続維持、定期実行（30秒間隔）による処理継続、system_logsテーブルによる動作監視【拡張性】100ホスト対応、プロキシ数拡張（上限未定義）、データベース容量は運用環境依存
