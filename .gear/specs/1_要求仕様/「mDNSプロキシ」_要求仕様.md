# 「mDNSプロキシ」_要求仕様

> バージョン: 2 | 更新日時: 2026/06/11 09:21:00

## プロジェクト概要・背景
mDNSはネットワークセグメントを超えないため、各セグメントに配置した「mDNSプロキシ」同士でmDNSテーブルを相互交換し、セグメントを跨いだ名前解決を実現する。

## プロジェクトの目的
各ネットワークセグメントに「mDNSプロキシ」を配置し、相互にmDNSテーブルデータを交換してセグメント跨ぎのmDNSを提供する。

## 対象プラットフォーム
- ハードウェア: Raspberry Pi Pico（最新モデル想定、Pico標準のWi‑Fi接続を想定）
- OS/実行形式: MicroPython（Pico版）、Windows用EXE（PyInstallerで生成）
- ライセンス: MIT

## 機能要件
- アーキテクチャ
  - 実装言語: MicroPython（Pico）、Python（Windows EXE）
  - DB: SQLite3

- データベース構成
  - static_DB: 管理者が「検索ホスト名.ini」に記載した対象ホスト名（またはFQDN）を保持
  - self_DB: static_DB のホスト名について名前解決で得られたIP（IPv4/IPv6）を mDNS hosts 形式で保持・更新
  - other_DB: 外部 mDNSプロキシから受信した self_DB データを格納
  - other_proxy_DB: 外部 mDNSプロキシの IP アドレスとトークンを記録・更新
  - merged_DB: self_DB と other_DB をマージした最終 mDNS レコード（hosts 形式）として利用

- 設定ファイル
  - 検索ホスト名.ini
  - システム.ini
    - 外部 mDNSプロキシ IP アドレス（固定リスト）
    - 発信トークン接頭語（デフォルト: `mDNSProxy_`。実際のトークンは 接頭語 + MAC アドレス）
    - 実行間隔（デフォルト: 30秒）

- ネットワーク／プロトコル
  - mDNS 通信（ローカルの mDNS 応答）は従来通り UDP/5353 を使用
  - プロキシ間のデータ共有は HTTP（TCP/80）を使用（トークン認証を付与）

- 動作フロー（実行間隔でループ）
  1. mDNSプロキシ発見: 設定された固定IPおよび受信した発信トークン（接頭語+MAC）による発見
  2. static_DB の共有: 発見した全プロキシ間で最新タイムスタンプの static_DB を共有・更新
  3. self_DB の更新: 最新の static_DB を元に名前解決を行い self_DB を更新
  4. other_DB の取得: 外部プロキシから self_DB 相当データを取得・other_DB を更新
  5. mDNS 応答: self_DB と other_DB をマージして merged_DB を作成し、mDNS レコードとして応答。Windowsリゾルバ等の挙動において、ユニキャスト応答だけでは無視・破棄されるケースに対応するため、既存のソケットによる送信元ポート5353でのユニキャスト応答と並行して、mDNS標準のマルチキャスト（224.0.0.251:5353）に対しても同一の応答を同時にブロードキャスト（マルチキャスト送信）する。

- マージルール
  - 重複が発生した場合は「タイムスタンプ最新優先」

## 非機能要件
- 認証・暗号化
  - 当面は「トークンのみ」で認証（発信トークンを HTTP ヘッダ等で送信）
  - 将来的な強化（TLS や有効期限管理）は別途検討（現時点では未導入）

- データ交換フォーマット
  - 決定事項: レコード単位JSON送信（HTTPでJSONペイロード）
  - 実装方法: HTTP（TCP/80）で各レコードをJSON化して送受信し、受信側でother_DBへ反映

- TTL と古いレコード削除
  - TTL 初期値: 120 秒（管理者が system.ini で変更可能）
  - ガベージコレクションは定期実行（実行間隔に紐づく）で TTL 切れのレコードを削除

- 発見方式
  - 固定リスト（システム.ini）＋受動的な発信トークン受信の併用

- 実行間隔・同期頻度
  - デフォルト 30 秒でループ
  - 適応的バックオフは現時点では不要（将来的検討）

- エラーハンドリング
  - 外部プロキシ取得失敗時は「スキップ」して次サイクルへ（エラーは簡易ログに記録）
  - 再試行は次サイクルで行う

- ロギング・監視
  - ログはローカルに簡易記録（ファイル）。リモート監視/集約は不要
  - ログレベルは INFO/ERROR を想定

- 管理インタフェース
  - CLI（対話メニュー）での管理・設定変更を基本とする
  - Web UI / REST 管理 API は現時点では不要

- テスト要件
  - まずは単体テストを実装（主要機能の挙動確認）
  - 統合テストは後段で追加（ネットワーク混在環境での検証）

- パフォーマンス要件
  - サポート想定ホスト数: 100 ホスト
  - DB 上限: 無制限（運用環境に依存、Pico などはストレージ制約に注意）
  - レイテンシ: mDNS の応答特性に準拠すること（実運用での許容は暫定）

- Windows EXE
  - Pico 版と設定互換を保つ
  - 配布形態: インストーラあり（サービス化可）

## セキュリティ注意
- HTTP（TCP/80）＋トークンのみは盗聴・改ざんリスクがあるため、運用ネットワークの信頼性を担保するか、将来的に TLS 導入を推奨する。
